理解 ScrollView

首先要明确的是，scrollview 其实和普通的 view 并没有多大的差别，只不过给它加上了一些手势和约定。

我们知道，要让一个 scrollview 能够滚动的方法是设置它的 contentSize 的宽或者高或者同时比自己的大。

想一想为什么要这样做？

首先，scrollview 被苹果开发出来是为了通过滚动来显示比自己的 size 更多的内容，如果它的 contentSize 比自己都小，就没有滚动的必要了。

但实际上，scrollview 并没有滚动，当你在“拖动” scrollview 时，你其实是在改变 scrollview 的 bounds.origin。

那么什么是 bounds 呢？ bounds 和 frame 又有什么关系呢？

bounds 是一个 view 自身的坐标系，它的作用是规定它自身的内容从什么位置开始绘制。 frame 是一个 view 相对于自己父视图的位置。它们是完全不相关联的。

你已经知道的是，当你“拖动” scrollview 时， scrollview 的代理会收到scrollViewDidScroll: 的消息，在里面打印 scrollview 的 contentOffset 的值，这个值会随着你的“拖动”不停的变化。这时候，尝试打印一下 scrollview 的 bounds.origin 的值，你会发现，这个值竟然和前面的值一样。contentOffset 只是为了更好的说明情况给 bounds.origin 换了个名字而已。

contentSize 只是抽象的概念，它规定的其实是 scrollview 的 bounds.origin 能够变化的范围。

理解到这里，像其他的 scrollview 的特性，例如 bounces 等，都可以得到解释。

scrollview 还增加了一个特性，contentInset，它用来 scrollview 增加额外的滚动范围，其实就是把 bounds.origin 的变化范围扩大了。当你给 scrollview 设置 contentInset 的时候，你会发现，它的 contentSize 并没有改变。

一般情况下，你并不会通过直接设置 contentSize 来达到你“滚动”的目的，因为当你想让 scrollview 的所有内容都能够通过“滚动”的方式显示时，你很难计算子视图整体的大小， 因为子视图在当时有可能还没决定自己的大小。所以通常来说，你会用 scrollview 的子类，例如 tableview 和 textview。他们会根据自己内容的大小自动的设置 scrollview 的 contentSize，以便包裹所有内容。当然他们有自己不同的使用场景。

同样，当我们试图实现类似的功能时，我们该怎么做呢？

这里我们会用到 Auto layout。最佳实践是首先在 scrollview 上添加一个四周的 edge constraints 与 scrollview 间距为0的 content view，接下来，把你想要的子视图添加上去并且设置好相应的约束就可以了。

当我们实现上面的功能后，如果我们还想通过增加一些额外的滚动空间来实现例如防止键盘遮挡视图、下拉刷新或者上拉加载的功能时，我们就可以通过简单的设置 scroll view 的 contentInset 来实现，而不必从一开始就去关心 contentSize 的大小。

问题思考：scrollView 的代理有 scrollViewDidScrollToTop: 的方法来通知 scrollView 滑动到了顶部，想一想它的原理是什么？如何知道 scrollView 滑动到底部了呢？如果还要加上考虑 contentOffset 和 contentInset 呢？

Block 内存管理总结

前面只是大概的提了一下 block 还不够深入，这里通过回答几个经常碰到的问题来对 block 有个更深入的理解。

名词解释：

  • automatic variable、local variable、局部变量都代表一个东西，存活在栈上。

  • block literal (字面量)，例：^ { printf(“hello world\n”); }

问题1

通常来说，我们会把 block literal 赋值给声明的 block 变量，那么这些 block 都是什么类型呢？

以下规则适用于 MRC & ARC：

当block literal 被赋值给函数内部的局部变量时，一开始会被创建在栈上， isa 指针指向 _NSConcreteStackBlock,
当block literal 被赋值给 static 的变量或者全局变量时，会被创建在 text 区，isa 指针指向 _NSConcreteGlobalBlock，
特别的，当block 没有引用外部变量时，clang 会做一个优化，这时的 block 为 _NSConcreteGlobalBlock。

如果把 block 进行 copy （block_copy）操作后，该 block 会被 copy 到堆上，变成 _NSConcreteMallocBlock。

特别的，由于在 ARC 中对象声明都默认有 __strong 修饰，所以此时的 block 会被 copy 到堆上，这点需要注意。

问题2

我们都知道 block 会捕获外部的变量，那么捕获变量又有哪些规则呢？

以下规则适用于 MRC & ARC：
对于局部的基本数据类型的变量，block 会 copy 一份值，并且不可修改。
对于全局、静态、属性基本类型变量，blcok 和外部共享一份，可以修改。

有区别的地方：

对于局部的对象，non-arc 会copy一份指针，并且不会 retain；arc 会强引用（retain，引用计数加一），均不可修改。

对于属性对象 & self，arc下会被 retain，
而 non-arc 需要分情况：
如果为 stack block，那么就不会被 retain；
如果为 malloc block，就会被 retain。

(注：除了 NSString *可以声明为静态、全局的对象外，其他的对象都不行，字符串比较特殊，可以作为编译时常量)

问题3

如果我们修改捕获的变量，又该怎么办呢？

我们可以在 block 声明的前面加上 __block 修饰，此时，就可以修改被捕获的变量了，并且会影响外部的值。

只是要注意的是，只要 block 被 copy 后（无论是被强引用还是手动 copy），捕获的变量都会被 copy 到堆上，此时原来栈上的 __block 变量会被销毁。

问题4

什么时候会产生循环引用？如何解决呢？

简单来说，只要 self 持有了 block，而在 block 里又对 self 进行了捕获，那么就会导致循环引用，所以一般的栈上的 block 就不会引起循环引用。

解决方案就是打破循环引用，一般来说：
在 MRC 下，创建一个用 block 修饰的指向 self 的指针，在 block 里使用它。
在 ARC 下，创建一个用 weak 修饰的指向 self 的指针，在 block 里使用它。

参考链接
http://clang.llvm.org/docs/Block-ABI-Apple.html

iOS 内存管理基础

在开始讲解之前，先科普几个后面可能提到的名词：

  • LLVM & clang：Low Level Virtual Machine (LLVM) 是一个开源的编译器架构，它已经被成功应用到多个应用领域。Clang ( 发音为 /klæŋ/) 是 LLVM 的一个编译器前端，它目前支持 C, C++, Objective-C 以及 Objective-C++ 等编程语言。Clang 对源程序进行词法分析和语义分析，并将分析结果转换为 Abstract Syntax Tree ( 抽象语法树 ) ，最后使用 LLVM 作为后端代码的生成器。
  • MRC & MRR：Manual Reference counting，手动引用计数，因为需要在代码里面手动调用 retain、release 来管理内容，所以也被称为 Manual retain release。
  • ARC：自动引用计数，是一个编译器特性，遵循手动引用计数的规则，替开发者在编译时期添加合适的代码。
  • 持有对象：持有某个对象会让这个对象的引用计数加一。
  • 内存泄漏：对象使用完毕后未释放，结果导致一直占据该内存单元，直到程序结束。一般来说就是指向该对象的指针释放了，但是对象未释放。
  • 作用域：一段代码能够起作用的区域，一般来说，两个大括号内代表一个作用域。

MRC

对于一个 2015 年才开始接触 iOS 开发的新手来说，不知道 MRC 算是很正常的，因为从 Xcode 4.2 开始就默认开启 ARC 了。那么是不是说学习 MRC 对我们来说就没用了呢，当然不是，学习 MRC 能帮助我们更好的了解 Objective-C 的内存管理机制，写出 bug-free 的代码。

我们首先要知道引用计数的规则：

  • 你持有你创建的对象。
  • 你可以通过 retain 来持有某个对象。
  • 在你不需要某个对象时，你必须通过 release 释放它。
  • 你不能释放你并不持有的对象。

下面我来逐一解释一下，在 MRC 环境中：

当你用以 alloc/new/copy/mutableCopy 开头的方法创建对象时，首先会为该对象分配内存，然后把该对象的引用计数会加一，通常来说，你还会声明一个指向该对象的指针，此时你就持有该对象。

在 Apple 的引用计数的底层实现中，用一张哈希表来管理对象的引用计数，当你调用对象的 retain 方法时，首先找到哈希表对应的位置，对象的引用计数会加一，此时你就持有该对象。

对象在离开声明它的那个 Scope 后，或者说你不再需要某个对象时，必须将它销毁，你不能手动调用 dealloc 方法，而是调用 release 方法减少引用计数，当引用计数为 0 时，对象会自动调用 dealloc 方法销毁对象，释放内存。

你不能对你并不持有的对象调用 release，否则程序可能会崩溃。

最后说一下 autorelease，对象通过调用 autorelease 方法注册到上下文中的 NSAutoreleasePool 时，当自动销毁池被销毁后，对象也会被自动的销毁，在 iOS 程序中，由 runloop 维护了一个自动销毁池，所以我们不用创建。在其他地方，我们可以通过 NSAutoreleasePool 类，或者 @autorelease { // code insert here } 创建自动销毁池。

ARC

为了让 iOS 程序员更多的关注应用的实现，而不是在内存管理上纠结，Apple 就推出了 ARC，这里还有专门的文档帮助你从 MRC 迁移到 ARC。

在 ARC 环境中，引用计数的规则没有发生改变，但是对象在声明时，必须用下列标识符之一标识：

  • __strong
  • __weak
  • __unsafe_unretained
  • __autoreleasing

默认情况下，所有对象的标识符为 strong。

当你声明一个被 strong 标识的指向一个对象的指针时，无论该对象以什么形式创建，该对象的引用计数都会增加一，你持有该对象。并且，当该对象离开作用域时，会自动插入调用对象的 release 方法的代码。

当你声明一个被 weak 标识的指向一个对象的指针时，对象的引用计数并不会增加，你就没有持有该对象，此时对象的引用计数为 0，对象会被自动释放，此时指向该对象的 weak 指针的值会被置为 nil。

unsafe_unretained 和 weak 有一点不同的是，当对象被释放后，指针不会被置空，而是继续指向已经被释放的内存空间，形成悬垂指针，所以是不安全的。

通常来说你不用手动标识 __autoreleasing，编译器会自动帮你处理。

Block

在平常的 iOS 开发的内存管理中，可能最难的就是 block。

首先 block 是一种对象，但它比较特殊的是，它一开始的时候是被创建在栈上的。

在 MRC 环境中，我们声明一个 block，如果该 block 没有引用外部的对象，那么这个 block 的类型为 NSConcreteGlobalBlock，如果该 block 引用了外部对象，那么该 block 的类型为 NSConcreteStackBlock，因为它本身在栈中，所以不会持有外部的对象。

你可以通过 copy 操作来将 block 复制到堆中，此时 block 的类型为NSConcreteMallocBlock，此时 block 会持有外部对象。

在 ARC 环境中，因为默认情况下对象指针被 __strong 标示，所以大多数时候创建的 block 都会被 copy 到堆中。

循环引用

如果两个对象互相持有，那么就会发生循环引用的悲剧，这时，我们只需要将不太重要的那个设置为标识为 __weak，循环引用就会被解开，内存就可以正常释放了。

问题

  • 为什么在 ARC 环境下我们还需要 autoreleasepool ?

在 cocoa 中，每个线程都会有一个 runloop，每个 runloop 有一个 autoreleasepool 栈，每次 runloop 结束后，这个栈会被销毁。

类方法返回的实例都会被标记为 autorelease，它会被自动添加到 pool 中，直到 pool 被销毁。
