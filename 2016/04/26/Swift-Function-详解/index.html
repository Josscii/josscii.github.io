Swift Function 详解

Swift 中的方法声明随着 Swift 的版本一直在改进，下面我们就来看看 Swift 中方法的演进史吧！

在开始讲解之前，我们先熟悉几个名词：

  • 函数名（function-name），就是函数的名字，上面的 myFunction 就是函数名
  • 函数签名（function-signature），函数签名和函数名不同，函数签名由参数列表和返回值组成
  • 函数类型（function-type），比如上面的 myFunction 的函数类型就是 (parameterType) -> returnType
  • 外部参数名（external-parameter-name­），外部参数名是为了达到和 OC 的函数兼容的目的，作用仅仅是为了让使用这个方法的人更清晰的了解传入的这个参数是什么意思。
  • 内部参数名（local-parameter-name­)，这和其他语言一样用来代替传入参数，使我们能够在内部拿到传入的参数。
  • 因为内外部参数名被放在参数的前面，所以还有 parameter-label 一说，为了说清楚某些情况，我们下面也会使用到它。

基本函数知识

Swift 方法既可以写成简单的没有参数外部名的 c 函数，又可以写成 OC 那种复杂的内外部参数名都有的函数。我们先来看一个例子：

func myFunction(param1: parameterType, param2: parameterType) throw -> returnType {
    // function body
}

首先我们要知道的是，在 Swift 中，在每个参数前只有一个 parameter-label 修饰时，第一个参数的外部参数名默认是被忽略的，而其他的参数的外部参数名和内部参数名相同，举个例子：

enum SongType: String {
    case Happy
    case Sad
}

// 1
func singASong(songName: String, songType: SongType) {
    print("Singing a \(songType) song named \(songName)")
}

我们会这样调用这个函数:

singASong("Happy Birthday", songType: .Happy)

我们看到调用这个函数时，第一个参数并没有外部参数名，而第二个参数有外部参数名，并且之前我们也把它作为内部参数名使用了。

注意：是上面那条规则只适用于普通函数，构造函数不适用。

为什么 Swift 函数会默认有这样的特性呢？你应该也发现了，这其实就是为了和 OC 的行为表现得一样。

因为我们在用 OC 写这样一个函数时通常像这样写：

- (void)singASong:(NSString *) songtype:(SongType)songtype;

如果我们想要给第一个参数加上外部参数，或者给其他参数去掉外部参数，你可以像下面这样做：

// 2
func singASong(songName name: String, _ songType: SongType) {
    print("Singing a \(songType) song named \(name)")
}

我们会这样调用这个函数:

singASong(songName: "Writting Code", .Sad)

哦？这两个函数竟然可以同时存在? 怎么回事？先不要着急，我们试试给第一个函数换一个返回值试试：（如果一个函数没有指明函数返回值类型，那么默认返回 void ）

// 3
func singASong(songName: String, songType: SongType) -> String {
   print("Singing a \(songType) song named \(songName)")
    return "no error"
}

接下来我们给第一个函数换一个函数名字：

// 4
func singASon(songName: String, songType: SongType) {
    print("Singing a \(songType) song named \(songName)")
}

此时并没有报错，我们来想想 Swift 是怎么区分函数的呢？

第一个函数的签名是， (:songType:) -> void
第二个函数的签名是， (songName::) -> void
第三个函数的签名是， (:songType:) -> String
第四个函数的签名是， (:songType:) -> void

第一个函数和第二个函数可以并存是因为，两个函数的参数列表的外部参数名不同，

第一个函数和第三个函数可以并存是因为，两个函数的返回值不同，

当然函数名不同的函数肯定可以并存，就像函数一和四。

所以总结下来，函数是根据函数名和函数签名来区分函数的。

另外，OC 里面能够通过字符串生成一个 SEL 类型，这个字符串通常用来代表某个函数。为了兼容 OC，Swift 也支持这样的方式。

class Demo: NSObject {
    let button = UIButton()

    override init() {
        super.init()
        // 在 swift 2.0 之前，你可以像下面这样做：
        // button.addTarget(self, action: "buttonTapped:", forControlEvents: .TouchUpInside)
        // 从 swift 2.2 开始，直接通过字符串生成 Seletor 的方式已经被抛弃了，取而代之的是下面这种方式
       button.addTarget(self, action: #selector(Demo.buttonTapped(_:)), forControlEvents: .TouchUpInside)
}

    func buttonTapped(sender: UIButton) {
        print("button tapped")
    }
}

但是要注意的是，如果函数参数存在 OC 和 Swift 不兼容的部分，比如元组、枚举等，那么这个函数就无法转化成 Selector。

关于参数的可变性

在 Swift 2.2 及其之前，函数的参数默认是被 let 修饰的，并且传入的参数只是原变量的一个 copy。

所以这个参数没办法修改，像下面这样是不行的：

func add(a: Int) {
    a += 1
}

要想这个参数可变，你可以在参数前用 var 修饰，这样就可以愉快的修改参数的值啦：

func add1(var a: Int) {
    a += 1
}

var a = 0

add1(a)

a = 0

要想影响原变量的值，你需要在参数前用 inout 修饰：

func add2(inout a: Int) {
    a += 1
}

var b = 0

add1(b)

// b = 1

规则看起来还是蛮简单的，可是在 Swift 开源过后，这个设定被大量的开发者诟病，所以就有了下列提议：

  • 0003-remove-var-parameters
  • 0053-remove-let-from-function-parameters
  • 0031-adjusting-inout-declarations

他们认为 var 和 inout 容易引起歧义，这些提议也被接收了，将在 Swift 3.0 中正式使用，所以 Swift 3.0 后规则将变成这样：

  • let 作为不能被显示的用作参数修饰符，传入的参数默认就是不可在内部被修改的。
  • var 作为参数修饰符被去掉，目前的设计是在参数被传入后立即用一个相同名字的变量把不可变的传入参数给 shadow 掉，所以看起来在里面可以改变参数的值，但是不影响外部。
  • inout 被改变位置，就像这样 func foo(x: inout T) -> U。
